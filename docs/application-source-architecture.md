# Application source architecture

## Objective

Make the complete application shell maintainable without changing the legacy
webOS runtime, adding browser dependencies, or rewriting the shared state in a
single risky migration.

## Chosen architecture

`app/source/` is the canonical source of the application coordinator. It is
split into ordered ES5 fragments by responsibility. `scripts/build-app.js`
concatenates those fragments deterministically into `app/app.js`, which remains
the ready-to-run browser artifact loaded by `app/index.html` and packaged in
the IPK.

The coordinator uses source concatenation rather than runtime modules, while
tested ES5 UMD views are loaded independently before it. Coordinator fragments
therefore keep the exact lexical scope and function-hoisting behavior of the
verified monolith; extracted views own their DOM and private focus state behind
explicit dependency objects.

## Source boundaries

1. `00-runtime.js`: IIFE entry, dependency bindings, shared state, and runtime
   initialization values.
2. `10-shell-home.js`: localization helpers, common DOM helpers, navigation,
   Home rows, focus, artwork, backdrop, and background theme behavior.
3. `20-search.js`: search keyboard, request coordination, result
   virtualization, focus, and view lifecycle.
4. `30-library.js`: library tabs, filters, recommendations, catalog grids,
   Watchlist and Playlists, navigation previews, and library refresh actions.
5. `40-settings.js`: settings integration and persistence adapters.
6. `42-server.js`: server connections, failover, discovery, and Plex activity.
7. `45-setup.js`: onboarding adapters, account access, and profile switching;
   `setup-view.js` owns setup/profile DOM rendering and consumes controller
   snapshots.
8. `46-server-actions.js`: server picker actions and server switching.
9. `47-settings-navigation.js`: settings lifecycle and language-editor actions.
10. `48-diagnostics.js`: diagnostics snapshot adapters and lifecycle wiring.
11. `50-detail.js`: media detail rendering, media preferences, metadata refresh,
   seasons, episodes, and detail navigation.
12. `60-player-controls.js`: player clock, progress, controls, skip markers,
   chapters, track settings, and playback summaries.
13. `65-player-subtitles-playback.js`: subtitle synchronization, seek/rebuild
   coordination, playback recovery, resume, and player lifecycle.
14. `70-input-bootstrap.js`: view closing, global key/pointer/wheel dispatch,
   Home loading, event registration, media element events, and bootstrap.

The ordering is part of the build contract. Files use numeric prefixes so the
dependency and hoisting order is visible without consulting build code.

## Generated bundle contract

- `app/app.js` is generated and must not be edited directly.
- `npm run build:app` rewrites it from the ordered source fragments.
- `npm run check:app-bundle` compares the generated bytes in memory with the
  checked-in bundle and fails when they differ.
- `npm run verify` executes the bundle check before lint, type-check, tests,
  baseline checks, and asset verification.
- ESLint and TypeScript validate the generated bundle, which is exactly what
  the TV executes. Fragment files are excluded as independent compilation
  units because the opening and closing IIFE intentionally span the ordered
  source set.

## Further extraction

The existing pure UMD modules remain the preferred destination for logic that
can be isolated from shared UI state. New behavior should first be implemented
as a tested module when practical, then wired from the appropriate source
fragment. The fragments are not permission to grow new monoliths: a fragment
approaching roughly 1,500 lines should be split again or have pure logic
extracted.

## Compatibility and release

The browser still downloads one ES5 `app.js`, preserving Chrome 53 behavior,
function hoisting, startup order, packaging, and offline installation. Node.js
is required only for development checks and rebuilding the bundle, not for
installing or running Ploff.
